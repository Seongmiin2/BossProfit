메뉴 원가 관리 DB 구조

이 프로젝트의 핵심은 메뉴와 재료를 바로 붙이지 않고, 사용량을 저장하는 RecipeItem 중간 테이블을 둔 것입니다.

Menu

PKid
메뉴 고유 ID
name
메뉴명, 예: 왕돈까스
category
카테고리, 예: 돈까스
price
판매가

RecipeItem

PKid
레시피 항목 ID
FKmenuId
어떤 메뉴의 재료인지
FKingredientId
어떤 재료를 쓰는지
quantity
해당 메뉴에 들어가는 사용량

Ingredient

PKid
재료 고유 ID
name
재료명, 예: 돼지고기
unit
단위, 예: g, ml, 개
unitPrice
단위당 가격

1. 왜 중간 테이블인가

메뉴와 재료는 여러 개가 서로 연결됩니다. 그런데 연결만으로는 사용량을 저장할 수 없어서 RecipeItem.quantity가 필요합니다.

2. 원가 계산 기준

메뉴 원가는 quantity * unitPrice의 합계입니다. 재료 단가가 바뀌면 API 계산 결과도 바뀝니다.

3. ERP와의 연결점

Menu는 상품, Ingredient는 원자재, RecipeItem은 제품별 소요 자재 구조로 설명할 수 있습니다.

API가 DB를 사용하는 흐름

1브라우저가 /api/menus/1/cost를 요청합니다.
2서버가 Menu와 연결된 RecipeItem을 조회합니다.
3각 항목의 Ingredient.unitPrice를 함께 읽습니다.
4원가, 마진, 마진율을 계산해 JSON으로 반환합니다.
