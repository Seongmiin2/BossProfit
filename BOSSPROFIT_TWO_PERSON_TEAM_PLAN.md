## 1. 운영 원칙

BOSSPROFIT은 프론트엔드와 백엔드를 완전히 분리해 개발하지 않는다. 두 명이 각각 하나의 사용자 가치 흐름을 끝까지 책임진다.

```text
담당 A: 사장님이 데이터를 입력하고 결과를 이해하는 흐름
담당 B: 시장 데이터를 수집하고 신뢰 가능한 전망을 만드는 흐름
공동: API 계약, 데이터 모델, 보안, 통합 테스트, 배포
```

이 문서에서 담당 A는 제품 책임자를 겸하는 사용자 본인, 담당 B는 팀원으로 표기한다. 실제 역량에 따라 이름만 교체할 수 있다.

## 2. 담당 A — 제품·매장 운영 흐름

### 책임

- 사용자 문제와 MVP 범위 결정
- UI/UX와 모바일 사용성
- 인증·온보딩·매장 데이터 입력
- 메뉴 원가·판매량·손익 계산
- 시장 전망과 자영업 119 결과를 사용자가 이해하는 화면
- 최종 인수 테스트와 발표 시나리오

### 담당 기능

1. 회원가입·로그인·로그아웃
2. 매장 생성과 온보딩 진행 상태
3. 재료 등록 단계형 UI
4. 구매가격 입력 UI
5. 메뉴·레시피 등록과 원가 미리보기
6. 판매량 빠른 입력
7. 오늘 대시보드
8. 시장 가격 및 예측 결과 화면
9. 내 가게 영향 시나리오 화면
10. 자영업 119 채팅과 행동계획 화면
11. 모바일 하단 내비게이션 `오늘 | 입력 | 시장 | 119`
12. Playwright E2E와 화면별 빈 상태·오류 상태

### 담당 코드 영역

```text
frontend/
bossprofit/accounts/
bossprofit/profit/ 중 매장·메뉴·레시피·판매·원가 서비스
```

### 핵심 완료 기준

- 신규 사용자가 모바일에서 도움 없이 첫 분석까지 도달한다.
- 로그인하지 않은 사용자는 개인 데이터를 볼 수 없다.
- 재료·메뉴·판매 입력값이 새로고침 후에도 유지된다.
- 금액과 원가 계산은 서버 결과와 일치한다.
- 360px 화면에서 잘림과 가로 스크롤이 없다.

## 3. 담당 B — 시장 데이터·예측·RAG 흐름

### 책임

- 외부 데이터 수집과 품질 관리
- 시장 품목 및 단위 표준화
- 가격 예측 데이터셋과 모델
- 데이터 누수 없는 평가
- 예측 결과와 근거의 버전 관리
- RAG 문서 수집·검색 기반

### 담당 기능

1. PostgreSQL과 개발 데이터 환경
2. `MarketItem`, `MarketPriceObservation`, `IngestionRun`
3. KAMIS 수집기와 증분 upsert
4. 도매시장 경락가격·거래량 수집
5. ASOS·농업기상·기상예보 수집
6. 품목·주산지·관측소 매핑
7. 데이터 품질·누락률 보고
8. Last Value와 Seasonal Naive baseline
9. Base Price Model
10. Weather & Supply Impact Model
11. rolling-origin OOF 잔차 생성
12. Residual Correction Model
13. 예측구간 보정과 Model Registry
14. RAG 문서 적재와 검색 평가

### 담당 코드 영역

```text
bossprofit/market/            # 신규 시장 데이터 Django app
bossprofit/ingestion/         # 필요 시 별도 수집 app
ml/                          # 학습·평가·서빙
data/ 또는 artifacts/        # Git에 올리지 않는 원본·산출물 규칙 포함
```

### 핵심 완료 기준

- 같은 원본과 코드 버전으로 같은 데이터셋을 재생성할 수 있다.
- 원본 응답 또는 원본 참조가 보존된다.
- `issued_at`, `valid_at`, `collected_at`이 구분된다.
- 무작위 시계열 분할과 미래 데이터 누수가 없다.
- baseline부터 단계별 성능이 별도로 측정된다.
- 성능이 악화되는 보정 단계는 품목·horizon별로 비활성화된다.

## 7. 협업 방식

### 브랜치

```text
main
feature/a-기능명
feature/b-기능명
integration/주차
```

한 브랜치에서 두 사람이 동시에 같은 파일을 수정하지 않는다.

### Pull Request

PR에는 다음을 포함한다.

- 해결하는 사용자 문제
- 변경한 화면·API·테이블
- 테스트 결과
- 스크린샷 또는 응답 예시
- 남은 제한
- migration 및 환경변수 여부

자기 담당 영역은 상대방이 리뷰한다. 보안·금액 계산·예측 결과는 반드시 상호 리뷰한다.

### 매일 15분 동기화

1. 어제 완료한 것
2. 오늘 완료할 한 가지
3. 상대방에게 필요한 계약 또는 데이터
4. 막힌 것

상태 공유보다 의존성 해소를 목적으로 한다.

### 주 2회 계약 리뷰

- API 요청·응답
- DB migration
- 단위와 금액
- 예측 결과 schema

## 9. 범위 방어

두 명이므로 아래 기능은 MVP 이후로 미룬다.

- 세무·급여·근태·전자결재
- 복잡한 재고와 자동 발주
- 광범위한 POS 연동
- 전체 KAMIS 품목 예측
- 범용 뉴스 검색
- 검증되지 않은 Transformer·LSTM 우선 적용
- LLM 직접 SQL과 자동 쓰기

추가 기능 요청이 나오면 기존 P0 티켓 하나를 제외하지 않는 한 일정에 넣지 않는다.

## 10. 첫 번째 통합 마일스톤

다음 흐름이 실제 데이터 또는 fixture로 한 번 연결되면 Week 1을 완료한 것으로 본다.

```text
로그인
→ 개인 매장 선택
→ 재료 이름 입력
→ KAMIS 후보 조회
→ 품목 연결
→ 구매가격 저장
→ 시장가격과 내 구매가격 비교
```

완료 조건:

1. 두 매장 사이에 데이터가 노출되지 않는다.
2. KAMIS API 키가 없어도 fixture로 E2E 테스트가 가능하다.
3. 가격 기준일·단위·출처가 화면에 표시된다.
4. 매핑 confidence가 낮으면 사용자 확인 전 계산에 사용하지 않는다.
5. 실패 원인을 `IngestionRun`과 API 오류 응답에서 확인할 수 있다.
